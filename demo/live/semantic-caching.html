Semantic Caching

LLM calls are slow and expensive. Valkey can cache responses by meaning, not just exact text. Walk through it step by step.

1

Create a cache index

This tells Valkey to build an HNSW vector index. When we store a prompt later, Valkey will index its embedding so we can search by meaning.

2

Ask Claude a question

We'll embed this prompt with Amazon Titan, check the cache (it's empty, so it'll miss), call Claude Haiku for an answer, then store the response + embedding in Valkey for next time.

"What is Valkey and what is it used for?"
3

Ask the same thing differently

This is a different string, but it means the same thing. Valkey will find the cached response because the embeddings are similar.

"Can you explain what Valkey is and what it's used for?"
4

Try something unrelated

This is a completely different topic. The cache should miss because the embeddings are far apart, proving the cache is smart, not just returning everything.

"What is the best pasta carbonara recipe?"

How it works

On every query:
1. Embed the prompt (Titan, 1536-dim)
2. FT.SEARCH the cache index for similar embeddings
3a. If cosine distance < 0.30 → cache hit, return stored response
3b. If no match → cache miss, call Claude, store result with HSET + EXPIRE
