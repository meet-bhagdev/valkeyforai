Session Internals & Inspection

Understand exactly what Strands stores in Valkey — the shape of Session, SessionAgent, and SessionMessage objects — and how to query, inspect, and manage session data directly.

The Three Storage Objects

Every Strands session backed by Valkey stores exactly three types of objects. Here's what each one looks like on disk:

Session — top-level metadata

# Key: strands:session:{session_id}
# Type: STRING (JSON)
{
  "session_id": "user-42",
  "created_at": "2025-06-01T10:00:00+00:00",
  "updated_at": "2025-06-01T10:05:32+00:00",
  "metadata": {}
}

SessionAgent — per-agent state

# Key: strands:session:{session_id}:agent:{agent_id}
# Type: STRING (JSON)
{
  "agent_id": "default",
  "session_id": "user-42",
  "conversation_manager_state": {
    "message_count": 4
  },
  "interrupt_state": null
}

SessionMessage — individual turns

# Key: strands:session:{session_id}:msg:{message_id}
# Type: STRING (JSON)
{
  "message_id": "3f8a1c2d-...",
  "role": "user",
  "content": [
    {"text": "My name is Alex and I'm building a RAG pipeline."}
  ],
  "created_at": "2025-06-01T10:00:01+00:00"
}

# Assistant message with tool use looks like:
{
  "message_id": "7b2e9f4a-...",
  "role": "assistant",
  "content": [
    {"text": "Nice to meet you, Alex! ..."},
    {"toolUse": {"toolUseId": "...", "name": "search", "input": {"query": "RAG pipeline"}}}
  ],
  "created_at": "2025-06-01T10:00:03+00:00"
}

Inspecting a Live Session

Using the ValkeySessionRepository from Cookbook 01:

from strands.types.session import Session, SessionAgent, SessionMessage

repo = ValkeySessionRepository(host="localhost", port=6379)
SESSION_ID = "user-42"
AGENT_ID = "default"

# 1. Read session metadata
session = repo.read_session(SESSION_ID)
print(f"Session created: {session.created_at}")
print(f"Session updated: {session.updated_at}")

# 2. Read agent state
agent_state = repo.read_agent(SESSION_ID, AGENT_ID)
print(f"Message count: {agent_state.conversation_manager_state}")
print(f"Interrupt state: {agent_state.interrupt_state}")

# 3. List all messages in order
messages = repo.list_messages(SESSION_ID, AGENT_ID)
print(f"\n{len(messages)} messages in session:\n")
for msg in messages:
    # content is a list of content blocks
    text_blocks = [b["text"] for b in msg.content if "text" in b]
    preview = text_blocks[0][:80] if text_blocks else "[tool use/result]"
    print(f"  [{msg.role:9}] {preview}")

Counting Keys and TTLs

def session_stats(repo: ValkeySessionRepository, session_id: str) -> dict:
    """Return a summary of what's stored for a session."""
    pattern = f"strands:session:{session_id}*"
    keys = repo.client.keys(pattern)

    stats = {"total_keys": len(keys), "messages": 0, "min_ttl": None}
    for k in keys:
        if ":msg:" in k:
            stats["messages"] += 1
        ttl = repo.client.ttl(k)
        if ttl > 0:
            stats["min_ttl"] = min(stats["min_ttl"] or ttl, ttl)
    return stats

print(session_stats(repo, "user-42"))
# {'total_keys': 7, 'messages': 4, 'min_ttl': 3541}

Deleting a Session

To clear all data for a session (e.g., on logout or explicit reset):

def delete_session(repo: ValkeySessionRepository, session_id: str) -> int:
    """Delete all Valkey keys for a session. Returns number of keys deleted."""
    pattern = f"strands:session:{session_id}*"
    keys = repo.client.keys(pattern)
    if keys:
        return repo.client.delete(*keys)
    return 0

deleted = delete_session(repo, "user-42")
print(f"Deleted {deleted} keys")
# Deleted 7 keys

Multi-Agent Sessions

A single session can hold state for multiple agents — each gets its own agent key and message list. This is useful for orchestrator/subagent patterns:

from strands import Agent
from strands.session.session_manager import RepositorySessionManager

repo = ValkeySessionRepository(host="localhost", port=6379)

# Two agents sharing the same session_id but different agent_ids
researcher_mgr = RepositorySessionManager(
    session_id="workflow-001",
    agent_id="researcher",
    session_repository=repo,
)
writer_mgr = RepositorySessionManager(
    session_id="workflow-001",
    agent_id="writer",
    session_repository=repo,
)

researcher = Agent(
    system_prompt="You are a research agent.",
    session_manager=researcher_mgr,
)
writer = Agent(
    system_prompt="You are a writing agent.",
    session_manager=writer_mgr,
)

researcher("Research the key benefits of Valkey for AI workloads.")
writer("Write a short blog intro about Valkey for AI.")

# Valkey now has separate message histories for each agent
# under the same session: workflow-001
researcher_msgs = repo.list_messages("workflow-001", "researcher")
writer_msgs = repo.list_messages("workflow-001", "writer")
print(f"Researcher: {len(researcher_msgs)} messages")
print(f"Writer: {len(writer_msgs)} messages")

Key pattern for multi-agent: strands:session:workflow-001:agent:researcher:messages and strands:session:workflow-001:agent:writer:messages are separate lists. The session metadata key strands:session:workflow-001 is shared.

Refreshing TTLs

To extend a session's lifetime on activity (sliding expiry):

def touch_session(repo: ValkeySessionRepository, session_id: str, ttl: int = 3600):
    """Reset TTL on all keys for a session — call on each user interaction."""
    pattern = f"strands:session:{session_id}*"
    for key in repo.client.keys(pattern):
        repo.client.expire(key, ttl)

# Call after each agent invocation to keep the session alive
touch_session(repo, "user-42")
