Production & ElastiCache

Deploy the Valkey session backend to Amazon ElastiCache, configure connection pooling and TLS, and apply production-ready TTL and eviction strategies for multi-tenant agent workloads.

Prerequisites

  • Completed Cookbooks 01 and 02
  • An ElastiCache for Valkey cluster (or use the local Docker setup for testing)
  • AWS credentials configured

Connecting to ElastiCache

The only change from local development is the connection parameters. The ValkeySessionRepository from Cookbook 01 works unchanged:

import ssl
import valkey

# ElastiCache for Valkey endpoint (cluster mode disabled)
ELASTICACHE_HOST = "master.my-valkey-cluster.abc123.use1.cache.amazonaws.com"
ELASTICACHE_PORT = 6379

# Production client with TLS and connection pooling
client = valkey.Valkey(
    host=ELASTICACHE_HOST,
    port=ELASTICACHE_PORT,
    ssl=True,
    ssl_cert_reqs="none",  # ElastiCache uses AWS-managed certs
    decode_responses=True,
    socket_connect_timeout=5,
    socket_timeout=5,
    retry_on_timeout=True,
    health_check_interval=30,
)

# Drop-in: pass this client to ValkeySessionRepository
class ValkeySessionRepository(SessionRepository):
    def __init__(self, client: valkey.Valkey, ttl: int = 3600):
        self.client = client
        self.ttl = ttl
    # ... same implementation as Cookbook 01

VPC access: ElastiCache clusters are not publicly accessible. Your Lambda function, ECS task, or EC2 instance must be in the same VPC and security group as the cluster. For local development, use an SSM port-forwarding tunnel.

SSM Tunnel for Local Development

To connect to ElastiCache from your laptop during development:

# Start the tunnel (requires a jump host EC2 instance in the same VPC)
aws ssm start-session \
  --target i-0594295b8314b037f \
  --document-name AWS-StartPortForwardingSessionToRemoteHost \
  --parameters '{
    "host":["master.my-valkey-cluster.abc123.use1.cache.amazonaws.com"],
    "portNumber":["6379"],
    "localPortNumber":["6379"]
  }' \
  --profile default \
  --region us-east-1

Then connect to localhost:6379 as normal — the tunnel forwards to ElastiCache.

TTL Strategy for Multi-Tenant Workloads

Different session types warrant different TTLs. Encode this in the repository constructor:

SESSION_TTLS = {
    "interactive": 3600,      # 1h — active chat sessions
    "background": 86400,     # 24h — async agent workflows
    "persistent": 2592000,   # 30d — long-lived user profiles
}

def get_session_manager(
    session_id: str,
    session_type: str = "interactive",
    agent_id: str = "default",
) -> RepositorySessionManager:
    ttl = SESSION_TTLS.get(session_type, 3600)
    repo = ValkeySessionRepository(client=client, ttl=ttl)
    return RepositorySessionManager(
        session_id=session_id,
        agent_id=agent_id,
        session_repository=repo,
    )

# Interactive chat agent
chat_agent = Agent(
    system_prompt="You are a helpful assistant.",
    session_manager=get_session_manager("user-42", "interactive"),
)

# Long-running background workflow
workflow_agent = Agent(
    system_prompt="You are a document processing agent.",
    session_manager=get_session_manager("job-8f3a", "background"),
)

Sliding Expiry on Activity

Reset TTLs on every agent invocation so active sessions don't expire mid-conversation:

def chat(session_id: str, message: str) -> str:
    repo = ValkeySessionRepository(client=client, ttl=3600)
    session_mgr = RepositorySessionManager(
        session_id=session_id,
        session_repository=repo,
    )
    agent = Agent(session_manager=session_mgr)
    response = str(agent(message))

    # Slide the TTL forward on every interaction
    for key in repo.client.keys(f"strands:session:{session_id}*"):
        repo.client.expire(key, 3600)

    return response

Eviction Policy

Configure ElastiCache to use allkeys-lru so the least-recently-used sessions are evicted first when memory is full — rather than throwing errors:

# Set via AWS CLI (one-time cluster configuration)
aws elasticache modify-replication-group \
  --replication-group-id my-valkey-cluster \
  --cache-parameter-group-name valkey7-allkeys-lru \
  --apply-immediately \
  --profile default \
  --region us-east-1

Lambda Deployment Pattern

For serverless deployments, initialize the Valkey client outside the handler to reuse connections across warm invocations:

import valkey
from strands import Agent
from strands.session.session_manager import RepositorySessionManager

# Module-level — reused across Lambda invocations
_client = valkey.Valkey(
    host=os.environ["ELASTICACHE_HOST"],
    port=6379,
    ssl=True,
    ssl_cert_reqs="none",
    decode_responses=True,
    socket_connect_timeout=3,
    socket_timeout=3,
)

def handler(event, context):
    session_id = event["session_id"]
    message = event["message"]

    repo = ValkeySessionRepository(client=_client, ttl=3600)
    session_mgr = RepositorySessionManager(
        session_id=session_id,
        session_repository=repo,
    )
    agent = Agent(session_manager=session_mgr)
    response = str(agent(message))

    return {"statusCode": 200, "body": response}

Monitoring Session Health

def session_health(repo: ValkeySessionRepository) -> dict:
    """Quick health check — count active sessions and total message keys."""
    all_keys = repo.client.keys("strands:session:*")
    session_keys = [k for k in all_keys if k.count(":") == 2]
    msg_keys = [k for k in all_keys if ":msg:" in k]
    return {
        "active_sessions": len(session_keys),
        "total_messages": len(msg_keys),
        "total_keys": len(all_keys),
        "valkey_ping": repo.client.ping(),
    }

print(session_health(repo))
# {'active_sessions': 42, 'total_messages': 318, 'total_keys': 445, 'valkey_ping': True}
